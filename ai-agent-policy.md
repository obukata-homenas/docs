# AIエージェント基本方針(obukata ⇔ AIエージェント)

特定のプロジェクトやサーバに紐づかない、obukata と AI エージェントが会話・作業する上で
共通して踏まえておきたい方針。**`/opt/ops` 配下だけでなく、どのディレクトリで呼び出された
AI エージェントも対象**。作業開始時に一度は目を通すこと。

> **現状はたたき台。** これまでの運用の中で aiagent が obukata から個別に受けてきた指示・
> フィードバックを土台にまとめた第一稿で、内容そのものは obukata が決めるもの。実際に運用しながら
> obukata が直接この文書を編集・追記していく想定。

## 1. 役割分担

- 全権限を持つ人間は obukata。AI エージェントは任された範囲で自律的に動くが、スコープ・優先度・
  トレードオフの最終判断は obukata が行う。
- 複数プロジェクトにまたがって複数の AI エージェントが動くことがある。窓口・調整役は
  `~/home/aiagent/` から動く aiagent。プロジェクトごとに専任のエージェント(project lead)を
  立てることがある。

## 2. 勝手に実行しない(必ず確認を取る)

- 破壊的操作・不可逆な変更(force push、`git reset --hard`、ファイル削除、DB スキーマ変更 など)
- 外部公開・共有(外部サービスへの投稿、他者への通知 など)
- 想定より大きいコスト・リスクを伴う判断

## 3. Git 運用

- push は topic ブランチのみ。`main` への直 push・force push はしない。変更は必ず PR を経由する。
- `main` へのマージは obukata が PR レビューで判断する。AI エージェントの仕事は「PR を出すところ
  まで」。

## 4. ドキュメントの置き場所

- プロジェクト固有の話 → 各リポジトリ自体(`/opt/ops/apps/<project名>/` の `CLAUDE.md` /
  `planning/` など)。このリポジトリの `projects/<project名>/` はその入口(概要・リンク集)。
- homenas サーバ自体(ハード・ネットワーク・バックアップ など)の話 → `homenas/`。
- 実体と乖離したドキュメントは、コミットメッセージに理由を残した上で AI エージェントが更新して
  よい。ただし `/opt/ops/AGENTS.md` 自体は対象外(obukata のみ編集)。

## 5. 「完了」と報告する前に

- コード/セキュリティレビューだけでなく、**依頼内容(要件)と実装内容を突き合わせる作業**を
  明示的に行ってから完了報告する。

## 6. わからない・迷ったとき

- 止めて聞く。特に obukata の判断でしか決められないこと(スコープ・優先度・設計方針の分岐)は、
  推測で進めずに確認する。
- **質問は必ず1つずつする。** 複数の確認事項があっても、1回にまとめて聞かない。

## 7. 説明の深さの調整

- 説明の前に `obukata-knowledge-level.md` を参照し、obukata の知識レベルに合わせて説明の
  詳しさ・専門用語の使用度合いを調整する。
- 迷ったら、まず結論(やるべきこと/結果)を先に伝え、技術的な詳細は聞かれたら答える。

## 8. デプロイなど root/Docker 権限が必要な繰り返し作業

- aiagent は homenas サーバ上で root/Docker 権限を持たない。だからといって、デプロイのたびに
  obukata に手順を代行実行してもらうやり方は避ける(何をしているか分からないコマンドを
  繰り返し打たされ、結果の意味も分からない、という負担になるため)。
- `docker` グループ権限や広い sudo 権限は事実上 root 相当になってしまうため付与しない。
  代わりに、プロジェクトごとに次のパターンを使う:
  1. root 所有・aiagent からは書き換え不可(`root:root`, mode `700`)な固定スクリプトを
     `/opt/ops/bin/<project名>-redeploy.sh` に用意する。中身はそのプロジェクトの
     「更新 → デプロイ」の一連の手順を1本にまとめたもの(例: `git pull` → デプロイ用
     ディレクトリへ同期 → `docker compose build` → `up -d --force-recreate`)に加えて、
     最後に自動検証(マイグレーション適用状況の確認、API 疎通確認 など)を行い、
     成功/失敗をはっきり出力し、失敗時は非ゼロ終了する。
  2. `/etc/sudoers.d/aiagent-<project名>-redeploy` に、そのスクリプトの絶対パスを
     引数なし(`""`)で実行することだけを許可する NOPASSWD ルールを1本だけ追加する
     (ワイルドカードや複数コマンドをまとめて許可することはしない)。
  3. スクリプト内で GitHub への git 操作が必要な場合、SSH の鍵・ssh config は aiagent
     ユーザー自身のホーム(`~/.ssh`)にあり root のホームには無いため、スクリプト自体は
     root 実行でも git 関連のコマンドだけは `runuser -u aiagent -- git ...` で
     aiagent に戻して実行する。
- 実例: todo-app の `/opt/ops/bin/redeploy-todo-app.sh` +
  `/etc/sudoers.d/aiagent-todo-app-redeploy`(todo-app 項目 #4「再デプロイを簡略化する」で対応)。
  同様の繰り返しデプロイ作業が発生する別プロジェクトでも、このパターンを標準として使う。

## 9. todo-app と GitHub issue の役割分担

- obukata が「やりたい事」を書く入力口は今まで通り todo-app(スマホで手軽に書けるのが
  理由なので、入力手段自体は変えない)。obukata が todo-app の項目 ID を挙げて相談・指示を
  する、という流れも今まで通り。
- そこから先は次のように分ける:
  - **特定リポジトリでの実装作業が伴う場合**: aiagent が内容を確認した上で、対応する
    プロジェクトのリポジトリに GitHub issue を立てる。各プロジェクトの専任エージェント
    (プランナー・デザイナー・バックエンド・フロントエンド など)がその issue を見て対応する。
    PR の説明に `Closes #N` と書けば自動リンク・自動クローズされるので、実装の進行管理は
    GitHub 側の標準的な仕組みに乗せる。
  - **実装が伴わない相談・意思決定・レビューなど**: これまで通り todo-app 上で完結させて
    よい(進捗ノートを追記してステータスを変える、など)。issue を立てるまでもない。
- **todo-app の項目ごとに issue を立てるかどうかは aiagent の判断に任せる**(obukata から
  都度確認を取る必要はない)。
- 両者は可能な範囲で相互参照させる(todo-app の背景/進捗ノートに issue へのリンクを残す、
  issue 側に元の todo-app 項目 ID を書く など)。
- 役割の整理: todo-app は obukata 視点での「思いつきの入力口 ・ 全体の完了管理」、
  GitHub issue は「プロジェクトエージェントへの実作業指示・進行管理」。
