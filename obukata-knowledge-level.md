# obukata の知識レベル(AI向け説明調整用)

AI エージェントが obukata に説明する際、詳しさ・専門用語の使用度合いを調整するための参考資料。
`ai-agent-policy.md` の §7 から参照される。

> **現状はたたき台。** これまでの会話から aiagent が書いた第一稿。実態とズレている部分は
> obukata 本人が直接書き換えること。

## 得意・馴染みがある領域

- 自宅サーバの構築・運用(OpenMediaVault、Docker/docker compose、Nextcloud、Immich、
  WireGuard、AdGuard Home、Nginx Proxy Manager など)を自分で手を動かして構築してきた。
  CLI/SSH 操作、sudo・docker コマンドにも抵抗がない。
- 「システム構成・ディレクトリ構造をどう分けるか」のような設計レベルの話は自分でも発案できる
  (例: docs リポジトリの `homenas/` + `projects/` 分割は本人の発案)。

## 深掘りしなくていい領域

- git の内部動作(rebase、amend、force-push の仕組みなど plumbing レベルの話)は、GitHub 上の
  PR 単位の理解で十分。「PR の説明文は直った/直っていない」のような結論だけ伝えれば良く、
  内部の仕組みの説明は求められない限り不要。
- 分散同期プロトコルの内部実装(例: CouchDB のチャンク分割・レプリケーション方式)のような、
  直接のアクション判断に繋がらない技術的詳細は、説明されても不要(むしろ冗長に感じる)。
  「結局どうすればいいか」を先に伝え、詳細は聞かれたら答える。

## 説明時の心がけ

- 結論・やるべきことを先に、技術的な経緯・内部実装の説明は後回し(聞かれたら答える)。
- 「大きな決定(設計方針・優先順位)」は一緒に考えたい人。「実装の細部」は基本的に任せたい人、
  というスタンス。
